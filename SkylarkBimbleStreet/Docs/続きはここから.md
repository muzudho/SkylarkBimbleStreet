# 続きはここから

## 現状

- `Game1.cs` は外部アセットなしの 1画面アクションパズル試作版。
- 現在は Stage 1 から Stage 10 まで実装済み。ステージセレクトから各面を直接選べる。
- ステージ定義は `Stages/stage-001.json` から `Stages/stage-010.json` までの JSON ファイルへ移行済み。`StageDefinitions.cs` は `StageLoader.LoadStages()` を呼ぶだけの入口。ステージ型は `Stage.cs`、障害物型は `Hazard.cs`、JSON DTO は `StageData.cs`、読み込みは `StageLoader.cs`。`SkylarkBimbleStreet.Editor` プロジェクトを追加済み。
- ステージセレクトでは、各カードに実ステージデータ由来のミニマップ風プレビューが出る。カード下部には宝石数、障害物数、クリア状態の小アイコン列が出る。左右移動時は選択カードが中央へ戻るようにカード列がスクロールする。カード拡大は中央へ戻ったあとだけ、補間付きで滑らかに行う。左右移動は非ループで、端では矢印が暗くなる。初期表示では Stage 1 が中央で選択される。クリア済みステージにはチェック印が付き、アプリ起動中にクリアした分はポーズからステージセレクトへ戻っても残る。現在は実行中のメモリ上だけで保持し、保存はしない。
- Xbox コントローラーとキーボードで、移動、ステージ選択、ポーズ、リトライ、ステージセレクト復帰を操作できる。
- ポーズメニューの4枚目のカードで、通常、色覚配慮、高コントラスト、モノクロ確認用の色パレットを切り替えられる。ゲーム進行、HUD、ポーズ、ステージセレクト、クリア画面の大きな構成色に反映される。紙吹雪やランク宝石など一部の演出色は綺麗さ優先で残している。
- 宝石、出口、障害物、ミス表示は、色だけでなく形でも区別しやすくしている。障害物はX印ではなく警告バー型にしている。開いた出口は横棒矢印ではなく、中央の2本線が左右に開いたような表示にしている。
- 全クリア時は宝箱風クリア演出が出る。ミス数と開始ステージに応じて金、銀、銅のクリアランクが変わる。
- プレイ中だけメモリ上で統計イベントを記録している。ステージ開始、ジェム取得、ミス、クリア、ポーズ、リトライ、ステージセレクト開始が対象。
- ミス後はスタート地点へ戻り、0.8秒だけ無敵になってプレイヤーが点滅する。
- `Docs/開発/開発日誌.md` に 2026年6月11日のライナーノーツを追記した。
- `Docs/設計/images/clear-treasure-chest.png` に宝箱風クリア画面イメージを追加した。
- `Docs/設計/images/stage-select-prototype.png` に Stage 5 追加後のステージセレクト画面イメージを追加し、README、プレイ説明書、2026年6月12日の開発日誌から参照するようにした。
- `Docs/設計/初期要件.md` に「技術を剥がしても残る遊びの核」を成果物の必須方針として追加した。`Docs/企画/ゲーム案メモ.md` にも企画評価の重視条件として追加した。
- 現在の試作の中心核を「障害物の危険圏に踏み込み、宝石を取ったらすぐ離脱する、短いリスクとリターンの管理」として整理し、`Docs/設計/初期要件.md` と `Docs/開発/アクションパズル試作.md` に [必須] 方針として追記した。
- 制限時間では急かさないこと、コントローラーを離して放置してもミスにならず、落ち着いて観察と判断ができることを良い点として [必須] 方針に追加した。
- 障害物はプレイヤーを追いかけて襲う敵ではなく、プレイヤーが危険圏に踏み込むことでミスが起きる盤面要素と整理した。リアルタイムだが、プレイヤー自身がリスクを取る瞬間を選ぶターン制に近い特徴として記録した。
- ステージ設計では、立ち止まって考えられる休憩地点や一気に抜けるコースを置き、目押し、レバー操作、反射神経などプレイヤーごとの得意を活かせる攻略余地を [必須] 方針として追加した。
- 全員が全ステージを簡単にクリアできる必要はなく、誰かには苦手でも誰かの得意が刺さるステージを用意する方針を追加した。ただし、理不尽ではなく、得意な人ならかっこよく突破できる見せ場として作る。
- `Docs/設計/初期要件.md` と `Docs/開発/アクションパズル試作.md` で、1ステージ内の方針とゲーム開始からクリアまでの構成方針を分けた。ゲーム全体では、自分の得意が活きるルートを選ぶことを核にし、ワールドマップ式分岐や自由ステージ選択寄りの構成を有力候補として記録した。

## 確認

- 直近のコード変更時点で `dotnet build .\SkylarkBimbleStreet.slnx` は成功。警告 0、エラー 0。
- 実プレイで、2面と3面の進行不能修正、クリア演出、ステージセレクト、ポーズ、コントローラー操作、クリアランク表示を確認済み。
- 旧Stage 5、現在のStage 6追加後、右側の縦壁と横壁が接続して通路を塞いでいたため、右下の縦壁とジグザグ横壁を短くして隙間を作った。プレイヤーサイズ込みの到達判定で全宝石と出口に到達可能なことを確認し、修正後にビルド確認済み。実プレイでも旧Stage 5、現在のStage 6をクリア確認済み。
- ウィンドウを閉じたあとプロセス終了まで長く感じる件は、アプリ本体の終了処理ではなく Visual Studio デバッグ/Hot Reload 注入と MonoGame DesktopGL の SDL/OpenGL 終了処理の組み合わせが濃厚。直接起動や Visual Studio 由来の環境を引き継がない起動では約 160ms で終了し、イベントログの新規エラーも出なかった。一方、VS デバッグ系モジュールが入ると `ntdll.dll` / `0xc0000008` の `APPCRASH` と Windows Error Reporting が記録される。実際に Hot Reload のチェックを外し、Release モードで「デバッグなしで開始」したところ、ウィンドウを閉じたあとすぐ終了した。しばらくこの運用で様子を見る。
