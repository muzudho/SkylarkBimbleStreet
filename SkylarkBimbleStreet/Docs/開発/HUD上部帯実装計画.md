# HUD上部帯実装計画

作成日: 2026-06-16

## 目的

ゲーム画面左上に固まっているHUDを、上部の専用ステータス帯へ整理する。
プレイ領域と情報表示領域を分け、宝石袋、符牒の破片、ミス表示、ステージ進行などを見やすくする。

## 現状

- `Game1.cs` の `DrawHud()` にHUD描画がまとまっている。
- 現在は符牒の破片、宝石袋、ミス表示が左上付近に集まっている。
- ステージ進行表示は右上寄りに描かれている。
- 内部解像度は `1920 x 1080`。
- ステージデータは `Stages/stage-*.json` にあり、ゲーム本体とエディターは同じ座標系を使っている。

## 方針

まずはステージJSONや当たり判定を変えず、HUDの見た目と配置だけを上部帯へ移す。
ゲーム本体の座標変換やステージデータ調整は、HUD帯の見た目を確認したあとに必要なら行う。

## 実装案

### 案A: 低リスク版

上部にHUD帯を描き、HUD要素だけを横並びに移す。

- ステージJSONは変更しない。
- プレイヤー、壁、病院、バス停、出口、障害物の座標は変更しない。
- 当たり判定は変更しない。
- `DrawHud()` の固定座標を整理する。
- 最初の実装として採用候補。

利点:

- 既存ステージが壊れにくい。
- 実装範囲が小さい。
- HUDの見やすさだけを先に確認できる。

注意点:

- 厳密にはゲーム面積を縮めたわけではない。
- 上部のゲームオブジェクトとHUD帯が視覚的に競合する可能性は残る。

### 案B: 本命版

上部HUD帯を確保し、ゲーム本体を下側のゲームビューポートへ収める。

- 内部ステージ座標は `1920 x 1080` のまま保つ。
- 描画時だけ、ゲーム本体を `HudBandHeight` より下の領域へ変換する。
- 当たり判定やJSONを一気に変えずに、画面上のプレイ領域を下げる。

利点:

- HUD帯とプレイ領域が明確に分かれる。
- HUDがゲームオブジェクトに重なりにくい。

注意点:

- 描画変換の影響で、矩形の見え方や細い線の太さが変わる可能性がある。
- エディター表示との差が出る可能性がある。
- ステージセレクト、ポーズ、クリア演出との重なり確認が必要。

## 推奨手順

1. `Game1.cs` にHUDレイアウト用の定数を追加する。
   - `HudBandHeight`
   - `HudPadding`
   - HUD各要素の幅や間隔

2. `DrawHud()` の先頭で、上部全幅のHUD帯背景を描く。
   - 既存パレットの `HudBackground` を使う。
   - 必要なら半透明にする。

3. HUD要素を横並びへ再配置する。
   - 左: 符牒の破片
   - 中央左: 宝石袋
   - 中央右: ミス表示、バッジ系の将来枠
   - 右: ステージ進行表示

4. まず案Aとして、HUD配置だけを変更する。

5. 見た目確認後、必要なら案Bへ進む。
   - ゲーム本体描画を下側ビューポートへ収める。
   - ステージJSONと当たり判定はそのままにする。

## 確認項目

- HUDが左上に固まって見えない。
- 符牒の破片、宝石袋、ミス表示、ステージ進行が一目で分かる。
- Stage 1 から数ステージで、上部の病院、バス停、壁などとHUDが見た目上競合しない。
- ステージセレクト、ポーズ、クリア演出がHUD帯と不自然に重ならない。
- ウィンドウを小さくしてもHUD要素が潰れすぎない。
- `dotnet build .\SkylarkBimbleStreet.slnx -v:minimal` が成功する。
